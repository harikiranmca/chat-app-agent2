# Specification for TTT-4: Build the Backend Chat API with MongoDB and Docker Integration Tests

## Summary

Extend the existing Express API in `apps/api` to expose endpoints for storing and retrieving chat messages in MongoDB. The API will validate incoming requests and return structured JSON responses. Integration tests will run against a real MongoDB instance via the existing Docker Compose setup.

## Goals

- Provide a working REST API for chat messages (create and list).
- Persist messages in MongoDB using the native `mongodb` driver (already installed).
- Validate request payloads and return clear error responses.
- Deliver integration tests that exercise the full HTTP-to-database path against a real MongoDB container.
- Ensure tests are self-cleaning and repeatable.

## Non-Goals

- Authentication or authorization.
- Chat rooms, channels, or multi-tenancy.
- WebSocket / real-time messaging.
- Pagination or filtering of messages (simple ascending list is sufficient).
- Production deployment configuration.
- Mongoose or any ORM (use the native `mongodb` driver already in `package.json`).
- Frontend integration (separate story).

## User Flow

1. Client sends `POST /messages` with `{ "sender": "Alice", "content": "Hello" }`.
2. Server validates the payload, inserts a document into MongoDB, and responds with the created message (including server-generated `id` and `createdAt`).
3. Client sends `GET /messages` and receives an array of all stored messages in chronological order.
4. Client can verify API health via `GET /health` (already exists).

## Functional Requirements

### Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | `/health` | Returns `{ "status": "ok" }` (already implemented in `apps/api/src/index.ts`) |
| POST | `/messages` | Creates a new message |
| GET | `/messages` | Returns all messages in ascending chronological order |

### POST /messages

**Request body:**
```json
{
  "sender": "string (required, non-empty after trim)",
  "content": "string (required, non-empty after trim)"
}
```

**Success response (201 Created):**
```json
{
  "id": "string (MongoDB ObjectId as hex string)",
  "sender": "string",
  "content": "string",
  "createdAt": "ISO 8601 timestamp string"
}
```

**Validation error response (400 Bad Request):**
```json
{
  "error": "string describing what is missing/invalid"
}
```

Validation rules:
- `sender` must be present and non-empty after trimming whitespace.
- `content` must be present and non-empty after trimming whitespace.
- Request `Content-Type` must be `application/json` (handled by `express.json()` middleware).

### GET /messages

**Success response (200 OK):**
```json
[
  {
    "id": "string",
    "sender": "string",
    "content": "string",
    "createdAt": "ISO 8601 timestamp string"
  }
]
```

Messages are sorted by `createdAt` ascending (chronological order). If no messages exist, returns an empty array `[]`.

### Message Document Schema (MongoDB)

Collection name: `messages`

```typescript
{
  _id: ObjectId,          // auto-generated by MongoDB
  sender: string,
  content: string,
  createdAt: Date         // stored as BSON Date, serialized as ISO string in responses
}
```

The `_id` field is mapped to `id` (hex string) in all API responses. The `createdAt` field is generated server-side at insertion time using `new Date()`.

### MongoDB Connection

- Connection string read from `process.env.MONGODB_URI` (default: `mongodb://localhost:27017/chat-app`), already established in `apps/api/src/index.ts`.
- Database name derived from the URI (default: `chat-app`).
- The `MongoClient` instance must be shared across the application (created once at startup, accessed by route handlers).

### Error Handling

- Invalid JSON body: Express returns 400 automatically with `express.json()`; add an error-handling middleware to return a structured `{ "error": "..." }` response.
- MongoDB connection failure at startup: log and exit (or fail health check).
- Unexpected errors in route handlers: catch, log, and return `500 { "error": "Internal server error" }`.

## Acceptance Criteria

1. Backend connects to a real MongoDB instance (verified by integration tests hitting the Docker container).
2. MongoDB connection string is read from `MONGODB_URI` environment variable (source: `apps/api/src/index.ts` line 11, already in place).
3. `GET /health` returns `200 { "status": "ok" }` (existing, preserved).
4. `POST /messages` with valid body persists a document to the `messages` collection and returns `201` with the full message object.
5. `POST /messages` with empty `sender` or empty `content` returns `400` with an error message.
6. `GET /messages` returns stored messages sorted by `createdAt` ascending.
7. Integration tests run against MongoDB in Docker using the existing `docker-compose.test.yml` (mongo:7 image).
8. Tests can be run with: `npm run test:integration` (from repo root; script already defined in root `package.json`).
9. Tests clean up their data between test cases (drop the `messages` collection or delete all documents in a `beforeEach`/`afterEach` hook).
10. No in-memory database or mock is used for integration tests; all tests hit the real MongoDB container.

## Edge Cases

- **Empty string after trim:** `{ "sender": "   ", "content": "hi" }` must be rejected (sender is effectively empty).
- **Missing fields:** `{ "content": "hi" }` (no sender) returns 400.
- **Extra fields in body:** Ignored; only `sender` and `content` are extracted.
- **Empty database:** `GET /messages` returns `[]`, not an error.
- **Malformed JSON body:** Returns 400 with structured error (Express `express.json()` error handler).
- **Very long strings:** No max-length validation required for this story (can be added later).
- **Concurrent inserts:** MongoDB handles concurrency; no application-level locking needed.

## Implementation Notes

### File Structure (recommended)

```
apps/api/src/
  index.ts              # Express app setup, middleware, MongoClient init, export app
  routes/
    messages.ts         # Router with GET /messages and POST /messages
  models/
    message.ts          # TypeScript interface + collection helper (getCollection)
  middleware/
    validation.ts       # Validation middleware for POST /messages (or inline)
    error-handler.ts    # Express error-handling middleware for JSON parse errors
  integration.test.ts   # Integration tests (expand existing file)
  index.test.ts         # Unit tests (keep existing health check test)
```

Alternatively, a flatter structure is acceptable given the small scope:
```
apps/api/src/
  index.ts              # App setup, routes, middleware, export
  messages.ts           # Message route handlers + validation
  integration.test.ts   # Full integration test suite
  index.test.ts         # Unit test (health)
```

### Key patterns from the existing codebase

- ESM modules (`"type": "module"` in package.json), import with `.js` extensions in TypeScript (source: `apps/api/src/index.test.ts` line 3).
- Vitest as test runner, supertest for HTTP assertions.
- `app` exported from `index.ts` for supertest to use without starting the server.
- Integration tests separated by filename pattern (`integration.test.ts`), excluded from unit test runs.
- `tsconfig.base.json` uses `"module": "ESNext"`, `"moduleResolution": "bundler"`, strict mode.

### Integration test approach

- Tests import the Express `app` and use `supertest` to make HTTP requests.
- A `MongoClient` is created in test setup pointing to the Docker MongoDB (`MONGODB_URI` env var set by the test environment or defaulting to `localhost:27017`).
- Before each test (or test suite), the `messages` collection is dropped or cleared.
- After all tests, the MongoDB client connection is closed.
- The existing `docker-compose.test.yml` uses `tmpfs` for `/data/db`, so data is ephemeral to the container lifecycle anyway, but tests still clean up between runs for isolation.

### Running tests

From the repository root:
```bash
npm run test:integration
```
This executes:
1. `docker compose -f docker-compose.test.yml up -d` (starts MongoDB)
2. `npm run test:integration -w apps/api` (runs vitest on integration tests)
3. `docker compose -f docker-compose.test.yml down` (stops MongoDB)
